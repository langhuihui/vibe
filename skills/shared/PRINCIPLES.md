# 核心原则（所有角色共享）

> 各角色 Skill 开头引用本文件即可，不再各自重复声明。

## 铁律1：证据优先于声明

- 任何完成声明必须有**新鲜的**验证证据支持
- 禁止依赖 Agent 自我报告（"已完成"、"测试通过"）
- 禁止"应该"、"可能"、"看起来"等模糊词汇
- 必须：识别验证命令 → 执行 → 读取完整输出 → 确认

## 铁律2：根因优先于修复

- 未找到根因前不得提出修复方案
- 禁止猜测修复、Workaround 绕过
- 必须先找根因，再按分析修复

## 铁律3：3次失败质疑架构

- 同一问题修复3次仍失败 → **STOP** → 上报技术总监/人类
- 禁止"再试一次""换个方法"，必须记录已尝试方法并升级

## 铁律4：禁止 Workaround

以下行为严禁使用：

- ❌ 脚本报错后手动创建空文件/假数据
- ❌ 跳过失败的步骤继续执行
- ❌ 注释掉失败的测试用例
- ❌ 用硬编码值绕过动态逻辑
- ❌ 捕获异常但不处理（空 catch）
- ❌ 修改输出路径而非修复逻辑
- ❌ 返回默认值而非修复逻辑

## 升级路径

```
开发专员 → 技术骨干 → 技术总监 → 人类
测试专员 → 测试总监 → 人类
产品经理 → 产品总监 → 人类
```

## 红旗信号 — 看到以下念头立即 STOP

- "快速改一下" / "先这样绕过"
- "试试改X看是否有效"
- "不太理解但先这样改"
- "这个很简单，不需要流程"
- "注释掉这个测试试试"

**STOP。返回根因分析，理解问题后再行动。**

## 编码原则（基于 Andrej Karpathy 观察）

> 来源：[andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills)  
> 这些原则直接针对 LLM 编程的三大常见问题：错误假设、过度复杂、副作用修改

### 原则5：先思考再编码 (Think Before Coding)

**问题：** LLM 经常在不确定时做出错误假设，隐藏困惑，不主动澄清

**要求：**
- **显式声明假设** — 不确定时要问，而非猜测
- **呈现多种解释** — 存在歧义时不要暗自选择
- **合适时推回** — 如果有更简单的方法，要说出来
- **困惑时停止** — 说明不清楚什么，请求澄清

### 原则6：简洁优先 (Simplicity First)

**问题：** LLM 倾向过度工程化、臃肿抽象、实现1000行而100行就够

**要求：**
- 不要添加未被要求的功能
- 不要为单一用途的代码创建抽象
- 不要添加未被要求的"灵活性"或"可配置性"
- 不要处理不可能的错误场景
- 如果200行可以变成50行，就重写它

**测试标准：** 资深工程师会说这过于复杂吗？如果是，就简化

### 原则7：手术式修改 (Surgical Changes)

**问题：** LLM 经常作为副作用修改/删除理解不足的注释和代码，即使与任务无关

**要求：**

**编辑现有代码时：**
- 不要"改进"相邻代码、注释或格式
- 不要重构没有问题的东西
- 匹配现有风格，即使你会采用不同方式
- 如果注意到无关的死代码，提及它 — 但不要删除

**当你的更改产生孤儿时：**
- 移除因**你的更改**而变得未使用的导入/变量/函数
- 不要移除预先存在的死代码，除非被明确要求

**测试标准：** 每一行更改都应直接追溯到用户请求

### 原则8：目标驱动执行 (Goal-Driven Execution)

**问题：** 指令式任务缺乏明确成功标准，需要持续澄清

**要求：** 将指令式任务转换为可验证目标

| 指令式（避免） | 目标驱动（推荐） |
|------------|--------------|
| "添加验证" | "为无效输入编写测试，然后让它们通过" |
| "修复 Bug" | "编写能复现它的测试，然后让它通过" |
| "重构 X" | "确保测试在重构前后都通过" |

**多步骤任务的做法：**
```
1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]
```

**核心洞察：** 
> "LLM 特别擅长循环直到满足特定目标...不要告诉它做什么，给它成功标准，看它执行"  
> — Andrej Karpathy

**强成功标准让 LLM 可以独立循环。弱标准（"让它工作"）需要持续澄清。**

## 设计原则（API 与代码）

以下原则指导接口设计、参数设计与错误处理，与上述铁律并行使用。

### 原则9：参数默认即是最优

- 默认参数值应代表**推荐用法**，零配置即可获得合理、安全、可维护的行为
- 避免"占位默认""随便填一个"；默认即最优，调用方无需查文档也能用对
- 新增参数时优先设好默认，再暴露可选覆盖

### 原则10：接口命名意图驱动，而非实现驱动

- 命名表达**做什么、解决什么问题**，不暴露内部实现细节（算法、存储、协议等）
- 好的命名：`getUserProfile`、`submitOrder`、`validateToken`；避免：`queryUserTable`、`postToQueue`、`callRpcX`
- 实现可换，意图稳定；意图驱动便于演进与替换实现

### 原则11：大声报错，且自带解药

- 错误信息**明确、可定位**：包含上下文（如参数、ID、操作）、失败原因、发生位置
- **自带解药**：在错误信息或文档中给出可执行建议（如"检查 X 配置""重试前请先 Y"），或提供错误码/链接指向排查步骤
- 禁止静默吞错、空 catch、仅打 log 不抛；禁止模糊报错（如"Something went wrong"）
