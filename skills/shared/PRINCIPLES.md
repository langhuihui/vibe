```markdown
# 核心原则（所有角色共享）

> 各角色 Skill 开头引用本文件即可，不再各自重复声明。

## 铁律1：证据优先于声明

- 任何完成声明必须有**新鲜的**验证证据支持
- 禁止依赖 Agent 自我报告（"已完成"、"测试通过"）
- 禁止"应该"、"可能"、"看起来"等模糊词汇
- 必须：识别验证命令 → 执行 → 读取完整输出 → 确认

## 铁律2：根因优先于修复

- 未找到根因前不得提出修复方案
- 禁止猜测修复、Workaround 绕过
- 必须先找根因，再按分析修复

## 铁律3：3次失败质疑架构

- 同一问题修复3次仍失败 → **STOP** → 上报技术总监/人类
- 禁止"再试一次""换个方法"，必须记录已尝试方法并升级

## 铁律4：禁止 Workaround

以下行为严禁使用：

- ❌ 脚本报错后手动创建空文件/假数据
- ❌ 跳过失败的步骤继续执行
- ❌ 注释掉失败的测试用例
- ❌ 用硬编码值绕过动态逻辑
- ❌ 捕获异常但不处理（空 catch）
- ❌ 修改输出路径而非修复逻辑
- ❌ 返回默认值而非修复逻辑

## 升级路径

```
开发专员 → 技术骨干 → 技术总监 → 人类
测试专员 → 测试总监 → 人类
产品经理 → 产品总监 → 人类
```

## 红旗信号 — 看到以下念头立即 STOP

- "快速改一下" / "先这样绕过"
- "试试改X看是否有效"
- "不太理解但先这样改"
- "这个很简单，不需要流程"
- "注释掉这个测试试试"

**STOP。返回根因分析，理解问题后再行动。**
```
