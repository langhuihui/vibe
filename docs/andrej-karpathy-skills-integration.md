# Vibe 集成 Andrej Karpathy Skills 说明

**更新时间：** 2026-04-15

## 更改内容

已将 [andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills) 的 4 条核心编码原则整合进 Vibe 框架：

### 1. 更新了 `skills/shared/PRINCIPLES.md`

新增了**编码原则（基于 Andrej Karpathy 观察）**章节：

- **原则5：先思考再编码 (Think Before Coding)**
  - 显式声明假设，不隐藏困惑
  - 呈现多种解释，不暗自选择
  - 合适时推回，困惑时停止

- **原则6：简洁优先 (Simplicity First)**
  - 不添加未被要求的功能/抽象/配置
  - 如果 200 行可以变成 50 行，就重写

- **原则7：手术式修改 (Surgical Changes)**
  - 不"改进"相邻代码、注释或格式
  - 每一行更改都应直接追溯到用户请求

- **原则8：目标驱动执行 (Goal-Driven Execution)**
  - 将"添加验证"转换为"写测试验证无效输入，然后让测试通过"
  - 给成功标准，而非步骤指令

### 2. 更新了 `README.md`

在"核心原则"章节中：
- 保留了原有的 4 条流程铁律
- 新增了 4 条编码原则（来自 karpathy-skills）
- 将原有的 3 条设计原则调整为"API 设计原则"
- 统一编号为 1-11 条原则

## 为什么整合？

### andrej-karpathy-skills 解决的问题

Andrej Karpathy 观察到 LLM 编程的三大常见问题：
1. **错误假设** — 模型在不确定时做出错误假设并继续执行
2. **过度复杂** — 倾向实现臃肿抽象，1000 行而 100 行就够
3. **副作用修改** — 修改与任务无关的代码、注释

### Vibe 的互补价值

Vibe 原有的铁律侧重**流程约束**（何时停止、如何升级），karpathy 的原则侧重**单次代码生成质量**。两者结合：

- **流程层面**：Vibe 的铁律确保整体质量保障（证据、根因、升级）
- **执行层面**：karpathy 的原则改善每个 Agent 的代码生成行为
- **协同效果**：开发专员生成更简洁的代码 + 技术骨干审查时有明确标准

## 使用方式

### 对开发者的影响

所有调用开发相关 Agent（开发专员、技术骨干）的场景，现在会自动遵循这 11 条原则。

**示例：**

**修复 Bug 时（修复模式）：**
1. 技术骨干分析问题 → 遵循**原则 5**（先思考再编码，显式声明假设）
2. 开发专员修复代码 → 遵循**原则 6**（简洁优先）+ **原则 7**（手术式修改）
3. 测试专员验证 → 遵循**原则 8**（目标驱动：写测试复现，确保修复后通过）

**开发新功能时（完整/快速模式）：**
- 设计阶段：技术骨干遵循**原则 5**（呈现多种方案，不暗自选择）
- 开发阶段：开发专员遵循**原则 6**（不过度工程）+ **原则 8**（先写测试定义成功标准）
- 审查阶段：技术骨干检查是否违反**原则 7**（是否有不必要的修改）

### 对 Skill 的影响

`skills/shared/PRINCIPLES.md` 是所有 Skill 的共享原则文件。每个 Skill 在开头都会引用：

```markdown
> 核心原则遵循 `skills/shared/PRINCIPLES.md`
```

更新后，所有 Agent 自动继承这 11 条原则，无需修改各个 Skill 文件。

## 验证效果的方法

### 预期改善

引入 karpathy 原则后，应该观察到：

1. **更少的不必要更改** — diff 中只出现请求的修改（原则 7）
2. **更简洁的实现** — 首次实现就很简洁，不需要重构（原则 6）
3. **更多澄清问题** — Agent 会在不确定时主动问，而非猜测（原则 5）
4. **更清晰的成功标准** — 任务自动转换为"写测试→让它通过"（原则 8）

### 检查清单

提交代码前，可以问自己：

- [ ] 是否有与任务无关的代码被修改？（违反原则 7）
- [ ] 是否添加了未被要求的抽象或配置？（违反原则 6）
- [ ] 是否有隐藏的假设未澄清？（违反原则 5）
- [ ] 是否可以用测试来定义成功标准？（未应用原则 8）

## 进一步定制

如果某些项目需要额外规则，可以在项目根目录创建 `.vibe/docs/代码规范.md`：

```markdown
# 项目代码规范

## 通用原则

遵循 `skills/shared/PRINCIPLES.md` 的 11 条原则。

## 项目特定规则

- 使用 TypeScript strict 模式
- 所有 API 端点必须有测试
- 错误处理遵循 `src/utils/errors.ts` 的模式
```

技术骨干在代码审查时会同时参考共享原则和项目规范。
